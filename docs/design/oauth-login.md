# OAuth Login Integration Design

Status: draft (research sketch)
Owner: TBD
Last updated: 2026-07-12

## 1. Context and Goals

Floway currently accepts only two credential shapes on the control plane:
username + password, or the `ADMIN_KEY` backdoor (empty username on
`POST /auth/login`). Operators who run Floway behind their own identity
provider (Authelia, Zitadel, Keycloak, Dex, an in-house OIDC service)
have no way to route dashboard logins through that IdP.

Goal: add a **third** login path that terminates at any OIDC-compatible
self-hosted OAuth server, without disturbing the existing password
path or the `ADMIN_KEY` backdoor.

Non-goals for v1:
- No public identity providers wired by default (no GitHub / Google
  hard-code). The abstraction is provider-agnostic; the first concrete
  implementation is a `GenericOidcProvider` fed by an OIDC discovery
  URL.
- No automatic account creation. First-login for an unknown subject is
  refused; the admin must pre-link the identity.
- No refresh-token storage. OAuth is used only to verify identity;
  Floway then issues its own opaque session token, and the OAuth
  access/refresh tokens are discarded.
- No breaking change to `x-floway-session`, the `sessions` table, or
  the API-key data path.

## 2. Current State (as of 2026-07-12)

Server-side opaque session tokens; no cookies, no JWTs, no signed state
anywhere in the auth stack.

- **Login route** — `POST /auth/login` accepts `{username, password}` or
  `{username: '', password: ADMIN_KEY}`. See
  `packages/gateway/src/control-plane/auth/routes.ts:10-39`. `resolveLoginUser`
  timing-safe-compares `ADMIN_KEY` and falls back to
  `verifyPassword(user.passwordHash, password)`. No `/auth/register`
  exists; new users are created by admins through `POST /api/users`.
- **Session token** — 32 random bytes → 64 hex chars, generated by
  `generateSessionToken()` in
  `packages/gateway/src/shared/session-tokens.ts`. Persisted in the
  `sessions` table (id, user_id, created_at, last_seen_at) — see
  `packages/gateway/migrations/0028_user_accounts.sql:20-26`.
  `last_seen_at` is touched by `SqlSessionsRepo.getByIdAndTouch`
  (`packages/gateway/src/repo/sql.ts:331-376`); there is no TTL column.
- **Middleware** — `packages/gateway/src/middleware/auth.ts:27-75`
  (`authMiddleware`) tries the `x-floway-session` header (or a
  `?session=` query on `/api/dump/keys/:id/stream` for EventSource
  compatibility), otherwise falls back to API-key extraction from
  `?key=` / `x-api-key` / `x-goog-api-key` / `Authorization: Bearer`.
  Session tokens are only accepted for paths under `/api/` or `/auth/`
  (data-plane routes reject sessions with 401).
- **Public-path allowlists** — `auth.ts:8-9`:
  - `PUBLIC_PATHS = ['/api/health', '/favicon.ico']`
  - `AUTH_VALIDATE_PATHS = ['/auth/login']`
  Any new unauthenticated OAuth endpoint MUST be added to
  `AUTH_VALIDATE_PATHS`, otherwise the middleware short-circuits with 401
  before the handler runs.
- **Password hash** — PBKDF2-SHA256, 100 000 iterations, 16-byte salt
  (`packages/gateway/src/shared/passwords.ts`). Encoded as
  `pbkdf2-sha256$<iters>$<salt_b64>$<hash_b64>`. The 100k ceiling is
  driven by Cloudflare Workers Web Crypto limits and is called out in
  the file's comments.
- **Permission model** — binary `users.is_admin` plus the orthogonal
  flag `can_view_global_telemetry`. User id 1 is the seed admin and can
  never be deleted or demoted (`packages/gateway/src/control-plane/users/routes.ts:76,111`).
- **Repo layer** — contracts in `packages/gateway/src/repo/types.ts:131-154`
  (`UsersRepo`, `SessionsRepo`); SQL impls in
  `packages/gateway/src/repo/sql.ts:227-376`; in-memory impls in
  `packages/gateway/src/repo/memory.ts:67-161`. The `Repo` composite
  interface is at `types.ts:367-382`.
- **Frontend** — Pinia store `apps/web/src/stores/auth.ts` persists the
  session token via `useLocalStorage('floway-token')`; the `user`
  object is refetched from `GET /auth/me` on every cold start.
  `apps/web/src/api/client.ts:6-13`'s `authFetch` attaches the token
  as `x-floway-session` and calls `clearAuth()` on 401. The router
  guard (`apps/web/src/router.ts:14-22`) redirects unauthenticated
  users to `/login`.
- **Existing PKCE helper** — `apps/web/src/lib/pkce.ts` implements
  `generatePkce()`, `deriveChallenge()`, `parseCallbackPaste()`,
  `stashPkce/peekStashedPkce/recallPkce/clearPkce`. Written for upstream
  provider onboarding (Codex, Claude Code) but the PKCE + state
  generation is directly reusable. The "paste callback URL" flow is
  **not** reusable for user login — see §4 (Redirect Handling).
- **Existing `Fetcher` abstraction** —
  `packages/provider/src/options.ts` exports the `Fetcher` type
  `(url, init, recordUpstreamLatency?) => Promise<Response>` and
  `directFetcher`. All new OAuth outbound requests should go through
  this abstraction so operator-configured proxies apply.
- **Not reusable** — `packages/gateway/src/control-plane/auth/github-device-flow.ts`
  is the **Copilot upstream** GitHub authorization (device flow, hard-coded
  client_id `Iv1.b507a08c87ecfe98`), not a user-login mechanism.

## 3. Design Decisions

Each decision below is fixed for v1. Where an alternative was considered
it is noted for reviewer context.

### 3.1 Provider abstraction

`packages/gateway/src/control-plane/auth/oauth/providers/` will host:

```ts
export interface OAuthProvider {
  readonly providerId: string;      // stable slug used in URLs and DB
  readonly displayName: string;     // shown on login page button
  authorizeUrl(input: {
    state: string;                  // signed opaque payload (see §3.5)
    codeChallenge: string;          // PKCE S256
    redirectUri: string;
  }): string;
  exchangeCode(input: {
    code: string;
    codeVerifier: string;
    redirectUri: string;
    fetcher: Fetcher;
  }): Promise<{ accessToken: string; idToken?: string }>;
  fetchUserInfo(input: {
    accessToken: string;
    idToken?: string;
    fetcher: Fetcher;
  }): Promise<{ subject: string; email: string | null }>;
}
```

V1 ships a single implementation `GenericOidcProvider` that reads
`<issuer>/.well-known/openid-configuration` at boot to discover the
authorization, token, and userinfo endpoints. This covers Authelia,
Zitadel, Keycloak, Dex, Ory Hydra, and any self-rolled OIDC-compliant
service — the announced target audience for v1.

*Rejected alternative*: hard-code endpoint URLs per-provider. Cheap for
GitHub/Google, but the point of v1 is to accept an arbitrary self-hosted
IdP, and forcing every operator to supply four URLs when discovery is
one URL is the wrong trade.

### 3.2 First-login behavior — admin whitelist

An OAuth callback carrying a `(providerId, subject)` that has no row in
`user_oauth_identities` is refused with HTTP 403 and a redirect back to
`/login?error=not-enrolled`. The login page renders a fixed message:
"This account is not enrolled for OAuth login. Ask an administrator to
link your identity."

Rationale:
- Current account creation is admin-only (`POST /api/users`); the
  OAuth path should not silently become an open-registration hole.
- The identity ↔ user mapping stays explicit and auditable.

Admins pre-link via a new endpoint `POST /api/users/:id/identities`
(see §5).

### 3.3 Binding entry point

Only via a new dashboard page `/dashboard/settings/identities`. A logged-in
user clicks "Link <provider>" to run one OAuth roundtrip; the callback
handler observes the caller already has a session and takes the "link"
branch instead of the "login" branch (branch key is carried in the
signed state, see §3.5).

*Rejected alternative*: auto-link when the callback's `email` matches an
existing user's username. Convenient but relies on the OAuth server
returning `email` and on that address matching the local username;
both are brittle and neither is a strong identity claim. v1 stays
manual; email-based auto-link can be added later behind an admin toggle.

### 3.4 Redirect shape — real HTTP 302

`GET /auth/oauth/:provider/callback` runs the full server-side flow and
issues a 302:

- On login success — 302 to `/oauth/handoff#session=<token>&return_to=<path>`.
  The frontend page `apps/web/src/pages/oauth-handoff.vue` reads the
  fragment, calls `useAuthStore().setAuth({token, user})` after hitting
  `GET /auth/me`, then `router.replace(returnTo || '/dashboard')`.
  Fragment (not query) so the token never appears in server logs or
  `Referer` headers.
- On link success — 302 to `/dashboard/settings/identities?linked=<provider>`.
- On failure — 302 to `/login?error=<code>` (never surfaces the raw
  provider error text; the specific reason is logged server-side).

*Rejected alternative*: issue an `HttpOnly` cookie in the 302 response.
Would require introducing cookies to a codebase that has consciously
avoided them, and would fork the middleware's session-source logic. The
fragment approach keeps the "session token lives in localStorage, sent
as `x-floway-session`" invariant intact — the OAuth callback is the
one moment the token appears in a URL, and only in a fragment.

### 3.5 State / CSRF — HMAC-signed state

The `state` parameter is not a random nonce alone. It is
`base64url(payload) + '.' + base64url(hmac_sha256(payload, key))` where
payload JSON is:

```json
{
  "nonce": "<32 hex chars>",
  "providerId": "<slug>",
  "intent": "login" | "link",
  "linkUserId": <int | null>,     // present iff intent="link"
  "returnTo": "<path or null>",
  "exp": <unix seconds>
}
```

- Key comes from a new required env `SESSION_HMAC_SECRET` (min 32
  bytes). Added to the `EnvGetter` reads in
  `packages/platform/src/env.ts` and to the platform-cf / platform-node
  bootstrap files that surface env vars.
- `exp` = now + 10 minutes. Rejected on expiry.
- `providerId` in the state must match the `:provider` URL segment on
  the callback; mismatch is a hard reject (prevents state-swap between
  concurrent multi-provider flows).
- `intent="link"` requires a live session belonging to `linkUserId`;
  otherwise reject.
- The client-side PKCE verifier lives only in `sessionStorage` via the
  existing helper — the server never sees or stores it.

### 3.6 Refresh tokens — not stored in v1

The Floway session issued after OAuth login is the source of truth for
"is this user allowed to keep talking to the dashboard." The provider's
access/refresh tokens are used exactly once (for the userinfo call
during callback) and discarded.

If a future feature needs to call provider APIs on behalf of the user
(e.g. read group membership periodically), refresh tokens can be added
as nullable columns on `user_oauth_identities` without a breaking
migration.

## 4. Schema Changes

One new migration: `packages/gateway/migrations/0043_user_oauth_identities.sql`.

```sql
CREATE TABLE user_oauth_identities (
  user_id       INTEGER NOT NULL,
  provider_id   TEXT NOT NULL,
  subject       TEXT NOT NULL,
  email         TEXT,
  linked_at     TEXT NOT NULL,
  PRIMARY KEY (provider_id, subject)
);
CREATE INDEX idx_user_oauth_identities_user
  ON user_oauth_identities (user_id);
```

Rationale:
- `(provider_id, subject)` is the natural key; the primary key uses
  that pair so lookups on callback (`WHERE provider_id=? AND subject=?`)
  hit the clustered index.
- `user_id` is not `UNIQUE` — a single Floway user MAY link multiple
  subjects (e.g. work + personal IdP). The secondary index covers
  "list all identities for user X".
- `email` is nullable and informational (some IdPs never expose email).
  It is NOT a login key.
- No foreign key constraint — matches the repo's existing convention
  (see `api_keys.user_id` in `0028_user_accounts.sql`). Cascade on user
  deletion happens in app code (see §5).

### Repo contract additions

`packages/gateway/src/repo/types.ts`:

```ts
export interface UserOauthIdentity {
  userId: number;
  providerId: string;
  subject: string;
  email: string | null;
  linkedAt: string;         // ISO 8601 UTC
}

export interface UserOauthIdentitiesRepo {
  getBySubject(providerId: string, subject: string): Promise<UserOauthIdentity | null>;
  listByUserId(userId: number): Promise<UserOauthIdentity[]>;
  link(identity: UserOauthIdentity): Promise<void>;   // INSERT OR ABORT on PK conflict
  unlink(providerId: string, subject: string): Promise<void>;
  deleteByUserId(userId: number): Promise<void>;      // called from users/routes.ts deleteUser
}
```

Add the interface to the composite `Repo` type. Implement in both
`packages/gateway/src/repo/sql.ts` (as `SqlUserOauthIdentitiesRepo`,
place alphabetically between `SqlSessionsRepo` and `SqlApiKeyRepo`) and
`packages/gateway/src/repo/memory.ts`.

### Cascade wiring

`packages/gateway/src/control-plane/users/routes.ts` — the existing
`deleteUser` sequence (`apiKeys.softDeleteByUserId → sessions.deleteByUserId
→ users.softDelete`) gains one call:
`userOauthIdentities.deleteByUserId(userId)` inserted between
`apiKeys.softDeleteByUserId` and `sessions.deleteByUserId`.

## 5. Gateway Route Blueprint

All new routes live in `packages/gateway/src/control-plane/auth/oauth/routes.ts`,
mounted from `packages/gateway/src/control-plane/routes.ts` before the
existing `/auth/*` chain. Zod schemas colocated in
`packages/gateway/src/control-plane/schemas.ts` or a new
`schemas.oauth.ts` if the block grows.

### 5.1 Unauthenticated

- `GET /auth/oauth/providers` — returns
  `{ providers: [{ id, displayName }, ...] }`. Sourced from a
  registry loaded at boot from env config (see §7). Added to
  `AUTH_VALIDATE_PATHS` and `PUBLIC_PATHS` alternatives — pick
  `AUTH_VALIDATE_PATHS` to match the pattern used by `/auth/login`.

- `POST /auth/oauth/:provider/authorize-url` — body
  `{ intent: 'login' | 'link', linkUserId?: number, returnTo?: string,
    codeChallenge: string }`. Returns `{ url, state }`. The client
  generates the PKCE verifier locally via the existing
  `apps/web/src/lib/pkce.ts:generatePkce`, sends only the challenge,
  and stashes the verifier in `sessionStorage`. Server builds the
  signed state (§3.5) and returns the fully-composed authorize URL.
  Added to `AUTH_VALIDATE_PATHS`. `intent="link"` requires a session;
  the middleware runs before this handler, so the check happens by
  gating that intent inside the handler (the endpoint itself must
  stay in `AUTH_VALIDATE_PATHS` because the "login" intent has no
  session yet).

- `GET /auth/oauth/:provider/callback?code=...&state=...` — full server
  flow:
  1. Verify HMAC + expiry of `state`.
  2. Match `state.providerId` with URL segment `:provider`.
  3. Look up the provider in the registry; 404 if unknown.
  4. Fail fast if `?code` is missing / `?error` is present (redirect
     to `/login?error=<code>`).
  5. Call `provider.exchangeCode({ code, codeVerifier: ???, ...})`.
     **Problem**: the verifier is in the browser's `sessionStorage` and
     the callback is a top-level 302 from the IdP that does not carry
     the verifier. Solution: PKCE verifier is NOT sent from the client
     — the server generates it in step (2) of the authorize-url
     handler and stores it in the state payload (HMAC-signed so
     un-tamperable). Update §3.5 payload accordingly during
     implementation. Note the trade-off in the code review.
  6. Call `provider.fetchUserInfo({ accessToken, idToken })`; require
     a non-empty `subject`.
  7. Branch on `state.intent`:
     - `login`: `getBySubject(providerId, subject)` → if null, 302
       `/login?error=not-enrolled`. Otherwise create a session for the
       matched user via `repo.sessions.create(userId)` (existing).
     - `link`: verify `state.linkUserId` matches an active session
       whose user is `linkUserId` (double-check to prevent stale
       state replay after logout); call
       `repo.userOauthIdentities.link({...})`. Handle PK conflict
       (subject already bound to a different user) by 302ing to
       `/dashboard/settings/identities?error=already-linked`.
  8. 302 as described in §3.4.

  Added to `AUTH_VALIDATE_PATHS`.

### 5.2 Authenticated (session or admin)

Mounted under the existing `/api/*` group so `authMiddleware` enforces
session presence. Admin-only endpoints go under the same
`adminOnlyMiddleware` chain used by the current `/api/users`.

- `GET /api/users/me/identities` — list caller's identities (session).
- `POST /api/users/me/identities/:providerId/:subject/unlink` — self
  unlink (session; must belong to the caller).
- `GET /api/users/:id/identities` — admin: list any user's identities.
- `POST /api/users/:id/identities` — admin: pre-link a subject to a
  user before the user has ever logged in. Body:
  `{ providerId: string, subject: string, email?: string }`.
- `DELETE /api/users/:id/identities/:providerId/:subject` — admin
  unlink.

The self-unlink endpoint MUST refuse if it would leave a user with no
login credential at all (no password hash AND no other identity AND
not user id 1) — otherwise a non-admin could lock themselves out. Same
guard applies to admin unlink for a non-self target.

## 6. Frontend Blueprint

### 6.1 Login page — `apps/web/src/pages/login.vue`

Below the existing username/password form and the "Leave blank +
ADMIN_KEY" note, render a "Continue with <displayName>" button per
provider returned by `GET /auth/oauth/providers`. Clicking the button:

1. `const { verifier, challenge, state: nonce } = await generatePkce()`
   using the existing helper.
2. `stashPkce({ ...verifier, state: nonce, providerId })`.
3. `const { url } = await api.auth.oauth[':provider']['authorize-url'].$post(...)`.
4. `window.location.assign(url)`.

Show an inline error banner if `?error=` is present on the URL when the
page loads (initial mount reads `router.currentRoute.value.query.error`
and maps to a friendly message).

### 6.2 New handoff page — `apps/web/src/pages/oauth-handoff.vue`

Router-registered path `/oauth/handoff`, added to the router's
"unauthenticated-allowed" list (mirror how `/login` is exempted from
the "requires auth" guard in `apps/web/src/router.ts:14-22`). On
mount:

1. Parse the URL fragment (`location.hash.slice(1)`) into
   `{ session, return_to }`.
2. `useAuthStore().setAuth({ token: session, user: null })`.
3. `await useAuthStore().refreshMe()` (existing hydrator that calls
   `GET /auth/me`).
4. `router.replace(return_to || '/dashboard')`.
5. On any failure (missing fragment, `/auth/me` returns 401), clear
   auth and `router.replace('/login?error=handoff-failed')`.

Render a minimal spinner while running; the whole thing typically
resolves in one paint.

### 6.3 New dashboard page — `apps/web/src/pages/dashboard/identities.vue`

Registered under the dashboard route tree (mirror how
`apps/web/src/pages/dashboard/users.vue` is registered). Shows:

- A table of the caller's linked identities
  (`GET /api/users/me/identities`).
- "Link <displayName>" buttons per configured provider that follow the
  same flow as §6.1 but with `intent="link"` and
  `linkUserId = useAuthStore().currentUser.id` in the request body.
- "Unlink" per row → `POST /api/users/me/identities/.../unlink`.

Add a nav entry pointing at this page in `dashboard.vue`.

### 6.4 Admin user management page — `apps/web/src/pages/dashboard/users.vue`

Add a "Linked identities" sub-panel to the existing user-detail dialog
(the component in `apps/web/src/components/users/`). Reuse the same
list/link/unlink UI as §6.3 but calling the admin endpoints from §5.2.

### 6.5 Client wiring

- `apps/web/src/api/client.ts` — Hono RPC client will pick up the new
  routes automatically from the `@floway-dev/gateway/app-type` re-export.
  No edits needed if the new routes are chained into the exported app
  type.
- `apps/web/src/lib/pkce.ts` — reuse `generatePkce`, `stashPkce`,
  `peekStashedPkce`, `clearPkce`. Recall: in this design the PKCE
  verifier travels *back to the server* via the signed state (see the
  §5.1 callback step 5 note), so the browser stash is only needed as a
  fallback / debug aid. Re-evaluate during implementation whether the
  browser needs to stash it at all — likely not.

## 7. Configuration Surface

Providers are configured via env at the platform layer (both CF and
Node targets), read through the `EnvGetter` contract in
`packages/platform/src/env.ts`.

New env vars:

- `SESSION_HMAC_SECRET` (required if any provider is configured) —
  min 32 bytes, hex or base64 accepted. Used for state signing (§3.5)
  and any future signed handoff.
- `FLOWAY_OAUTH_PROVIDERS_JSON` (optional) — a JSON array read at boot:
  ```json
  [
    {
      "id": "corp-idp",
      "displayName": "Corporate SSO",
      "issuer": "https://sso.example.com",
      "clientId": "floway",
      "clientSecret": "...",
      "scopes": ["openid", "profile", "email"]
    }
  ]
  ```
  Empty / unset ⇒ OAuth login disabled (login page shows no OAuth
  buttons; `GET /auth/oauth/providers` returns `[]`; all
  `/auth/oauth/:provider/*` routes 404).
- `FLOWAY_PUBLIC_ORIGIN` (required if any provider is configured) —
  the externally-reachable origin used to compose `redirect_uri`.
  Example: `https://floway.example.com`. Must match what the operator
  configured in the OAuth server's allowlist. This is separate from
  request `Host` on purpose — behind a reverse proxy the request
  origin is unreliable, and the redirect_uri MUST be byte-identical
  across authorize and token-exchange.

The `redirect_uri` sent to the IdP is always
`${FLOWAY_PUBLIC_ORIGIN}/auth/oauth/${providerId}/callback` — one path
per provider, no query parameters.

*Rejected alternative*: store provider config in D1 and manage via
dashboard. Simpler for operators but adds a control-plane surface (and
a way to lock oneself out by misconfiguring the provider that owns
one's session). Env-configured for v1; move to DB later if operators
ask.

### 9.2 Manual end-to-end (both platform targets)

Use a public OIDC playground (e.g. `https://demo.zitadel.cloud` or a
locally-run Keycloak container) as the IdP. Pre-register:

- Client type: web / OIDC / PKCE required.
- Redirect URI: `http://127.0.0.1:8788/auth/oauth/dev/callback` for
  CF-dev, `http://127.0.0.1:5174/auth/oauth/dev/callback` if using
  Vite's proxy (see `apps/web/vite.config.ts` proxy list — the OAuth
  callback path MUST be added to `wranglerProxiedPaths`).

Env for `pnpm run dev`:
```
SESSION_HMAC_SECRET=$(openssl rand -hex 32)
FLOWAY_PUBLIC_ORIGIN=http://127.0.0.1:8788
FLOWAY_OAUTH_PROVIDERS_JSON='[{"id":"dev","displayName":"Dev IdP","issuer":"...","clientId":"...","clientSecret":"...","scopes":["openid","email"]}]'
```

Steps:
1. Boot `pnpm run dev`.
2. Log in as admin via password.
3. Create a second user "alice", note her id.
4. `POST /api/users/<alice-id>/identities` with the subject you know
   your dev IdP will emit (get it from a first-round "not-enrolled"
   attempt's server logs).
5. Log out. Click "Continue with Dev IdP" on `/login`. Complete IdP
   flow. Land on `/dashboard` as alice, session in
   `localStorage['floway-token']`.
6. Log in as admin again via password. Go to Users → alice → Linked
   identities. Unlink. Repeat step 5 — should now redirect with
   `?error=not-enrolled`.
7. As alice logged in via password, visit
   `/dashboard/settings/identities`, click Link, complete IdP flow,
   return with `?linked=dev` and see the row.
8. Repeat 1–7 on the Node target: `pnpm run dev:node` with the same
   env plus `FLOWAY_DB_PATH`, `FLOWAY_FILES_DIR`, `ADMIN_KEY`, `PORT`.
9. `git diff main...HEAD --stat` for the completed feature branch
   should include the migration, gateway route files, repo additions,
   test files, and the four frontend files listed in §6. No changes
   outside those.

### 9.3 Rollback

- The migration is additive; rollback is `DROP TABLE
  user_oauth_identities; DROP INDEX idx_user_oauth_identities_user;`.
- Setting `FLOWAY_OAUTH_PROVIDERS_JSON=""` disables the whole surface
  without a code change or DB touch.
- The `sessions` table, password hash format, and middleware behavior
  outside the new whitelist entries are unchanged; any existing
  password-authenticated user keeps working.

## 10. Open Questions

Tracked here for follow-up, not blockers for review of the plan.

1. **PKCE verifier round-trip via state** — §5.1 step 5 has the
   verifier travel back to the server inside the HMAC-signed state
   payload. This is architecturally clean (no server-side pending
   table) but bloats every authorize URL by ~50–100 bytes. Confirm
   the target IdPs' state length limits (RFC 6749 says "arbitrary
   string," but some deployments cap at 512 or 2048 bytes).
2. **Provider display metadata** — where do icons come from? V1
   ships text-only buttons. Add icon URL to
   `FLOWAY_OAUTH_PROVIDERS_JSON` in a future revision if operators
   ask.
3. **Cookie handoff vs fragment handoff** — §3.4 chose fragment.
   Revisit if we discover any embedded-browser / OAuth-app-browser
   context where fragments are stripped.
4. **Multi-provider button ordering** — currently the JSON-array
   order. Consider promoting to a sort key if it becomes user-visible
   friction.
5. **IdP-initiated logout** — v1 does not propagate Floway logout to
   the IdP session, and vice versa. Fine for admin-controlled deploys;
   may want RP-initiated logout later.
6. **Rate limiting on the callback endpoint** — the callback is
   unauthenticated and does at least two upstream HTTP calls. If
   anti-abuse becomes relevant, add a per-IP token bucket keyed off
   the request origin (existing rate-limit infra: TBD, check repo).
